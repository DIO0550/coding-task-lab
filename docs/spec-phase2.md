# docs/spec-phase2.md — ゲーム進行・CPU対局エンジン仕様(Phase 2)

Phase 1 の点数計算エンジンの上に、**実際に4人麻雀を最初から最後までプレイできる**ゲーム進行エンジンと CPU 対局 AI を実装する。
対戦画面([docs/spec-match-screen.md](./spec-match-screen.md))は本エンジンを通じて「結果を表示するだけ」ではなく「実際に対戦する」体験を提供する。

## 関連仕様

- 和了判定・役判定・符計算・点数計算は Phase 1([docs/spec-phase1.md](./spec-phase1.md))のエンジンを再利用する。**重複実装しないこと。**
- 画面仕様は [docs/spec-match-screen.md](./spec-match-screen.md) を参照。本書はドメインロジック(ゲーム進行・CPU AI)のみを扱う。

## 牌山・配牌

- 牌は全136枚。赤ドラ「あり」設定時は 5m/5p/5s のうち各1枚を赤5(`0m/0p/0s`)に置き換える。
- 対局開始時に牌山をシャッフルし、王牌14枚を確保する。
- ドラ表示牌を1枚めくる。カン成立ごとに新ドラ表示牌を1枚追加する(最大5枚)。
- 各プレイヤーへ13枚を配牌する。親(起家)は東家から開始する。
- 残り山枚数は「牌山合計 − 王牌14枚 − 配牌 − ツモ済み枚数」で管理し、画面へ提供する。

## ターン進行

ゲーム進行は以下の状態機械として実装する。

1. 手番プレイヤーが山から1枚ツモる。
2. 手番プレイヤーは以下から行動を選択する。
   - ツモ和了(和了形かつ役ありの場合のみ)
   - カン(暗槓・加槓)
   - リーチ宣言(門前・テンパイ・持ち点1,000点以上・残り山4枚以上の場合のみ)
   - 打牌
3. 打牌に対し、他家は以下を宣言できる。優先順位は **ロン > ポン/カン > チー**。
   - ロン(和了形かつ役あり。フリテンでないこと)
   - ポン / 明カン
   - チー(下家のみ)
   - パス
4. 誰も宣言しなければ次の手番(下家)へ移る。
5. 和了または山が尽きるまで 1〜4 を繰り返す。

### リーチ

- リーチ宣言時に1,000点を供託へ出す。
- リーチ後は和了牌以外をツモ切りする(手替え不可)。
- リーチ者の和了時は裏ドラをめくり、Phase 1 エンジンへ `uraDora` として渡す。
- 一発・ダブルリーチの成立条件を管理し、エンジン入力へ反映する。

### フリテン

- 自分の捨て牌に和了牌が含まれる場合、ロン和了できない(ツモ和了は可)。
- リーチ後の見逃しは以降ロン不可とする。

### 和了処理

- 和了宣言時は Phase 1 エンジン(公開API)を呼び出して役・翻・符・点数を計算する。
- 点数移動(ツモの支払い内訳・ロンの放銃者支払い・本場加算・供託回収)はエンジンの結果に従って適用する。
- 画面側で点数計算ロジックを重複実装しないこと。

### 流局

- 山が尽きたら荒牌流局とする。
- テンパイ/ノーテンを判定し、ノーテン罰符(場3,000点)を移動する。
- 親がテンパイなら連荘(本場+1)、ノーテンなら輪荘(本場+1)とする。
- 供託は次局へ持ち越す。

### 連荘・輪荘・終了判定

- 親の和了: 親継続、本場+1。
- 子の和了: 親を下家へ移し、本場を0に戻し、次局へ進む。
- 東風戦は東4局まで、半荘戦は南4局まで。
- いずれかのプレイヤーの持ち点が0点未満になったら終局(トビ)とする。
- 終局時に持ち点で順位を決定する(同点は起家に近い方を上位)。

## CPU 対局 AI

CPU の手番では、人間の入力なしに自動で行動を決定する。レベルごとの要件は以下とする。

| レベル | 打牌 | 鳴き | リーチ | 押し引き |
|---|---|---|---|---|
| 入門 | 孤立牌・字牌から切る単純な牌効率 | 役に関係する明確な鳴きのみ | テンパイしたら常に宣言 | 考慮しない |
| 標準 | シャンテン数と受け入れ枚数を考慮 | 役と打点を考慮 | 打点・待ちの良さを考慮 | 他家リーチ時に現物を優先 |
| 上級 | 標準に加え打点効率を考慮 | 標準と同様 | 標準と同様 | 危険牌評価と降り判断を行う |

- 「CPU自動和了: あり」の場合、CPU は和了可能(役あり)なら必ずツモ/ロンを宣言する。
- 「CPU自動和了: なし」の場合、CPU はレベルに応じた判断で和了するかを選択してよい(初期実装では「あり」と同じ挙動を許容し、判断を `docs/assumptions.md` に記録する)。
- CPU の思考は同期的な純粋関数として実装し、`(卓の公開情報 + 自分の手牌) → 行動` の形にする。乱数を使う場合はシードを注入可能にしてテスト可能性を保つ。

## アーキテクチャ要件

- ゲーム進行の状態遷移は `(state, action) → state` の純粋関数(reducer 形式)で実装し、UI・React から分離する。
- 牌山シャッフルなど乱数を伴う処理は乱数生成器を注入可能にし、テストで再現できるようにする。
- Phase 1 エンジンとは公開APIを通じてのみ連携する。
- 外部ランタイム依存は追加しない。

## 受け入れ基準

- 対戦画面から対戦を開始すると、配牌・ツモ・打牌・鳴き・リーチ・和了・流局を経て局が進行し、東風戦/半荘戦を最後までプレイできること。
- 人間1人 + CPU3人の構成で、CPU が自動で打牌・鳴き・和了すること。
- 和了時の点数計算が Phase 1 エンジン経由で行われ、点数移動が [docs/spec-match-screen.md](./spec-match-screen.md) の「点数反映」規則と一致すること。
- ゲーム進行(配牌・ターン進行・鳴き優先順位・リーチ・フリテン・流局・連荘・終了判定)と CPU の行動決定に対するテストを自作し、すべて通ること。
- 解釈が分かれる事項は `docs/assumptions.md` に判断と根拠を記録すること。
