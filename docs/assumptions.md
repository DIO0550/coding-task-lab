# docs/assumptions.md — 仕様解釈の判断記録

`docs/spec-phase1.md` に未定義・曖昧な点があったため、以下の通り判断した。
各項目は「判断」と「根拠」を記載する。

## 1. 入力 `hand` の枚数(仕様の入力例が13枚である件)

- **判断**: `hand` は次の2形式を受理する。
  1. 和了牌を含む完成形: `hand.length === 14 - 3 × melds.length`(この場合 `winTile` と同種の牌が `hand` に含まれていなければ `invalid_input`)
  2. 和了牌を含まない形: `hand.length === 13 - 3 × melds.length`(この場合 `winTile` を加えて完成形として扱う)
- **根拠**: 仕様のコメントは「純手牌(和了牌含む)」だが、仕様・受け入れ基準の入力例の `hand` は13枚(副露なし)であり、和了牌 `5m` を加えて初めて14枚の完成形になる。例と文言が矛盾しているため、両形式を受理するのが最も安全と判断した。

## 2. `rules/cli-contract.md` が存在しない件

- **判断**: AGENTS.md から参照されている `rules/cli-contract.md` はリポジトリに存在しない。CLI 入出力契約は `docs/spec-phase1.md` の「入出力JSON契約」に従う。
- **根拠**: 仕様本文に入出力契約が完全記述されているため、それを唯一の契約とする。

## 3. CLI 層の配置

- **判断**: CLI エントリポイントは既存スケルトン通り `src/cli/calc.ts` に置く。`src/cli/` は `app/` と同様の「エントリポイント層」とみなし、stdin/stdout の I/O と JSON パースのみを行い、計算は `services/` に委譲する。
- **根拠**: `rules/architecture.md` のフォルダ一覧に CLI 層の定義はないが、既存スケルトンが `src/cli/` を確立している。ドメイン・サービス層での I/O 禁止(rules/coding.md)を守るため、プロセス境界の I/O は CLI 層に閉じ込める。

## 4. 本場・供託の入力フィールド

- **判断**: 入力 JSON にオプションフィールド `honba`(本場数, 0以上の整数, 省略時 0)と `kyotaku`(供託リーチ棒の本数, 0以上の整数, 省略時 0)を追加する。
- **根拠**: 仕様のスコープと受け入れ基準は「本場・供託を含む点数計算」を要求するが、入力契約に該当フィールドがない。既存の入力例を壊さないオプション拡張とした。
- 計算: ロンは放銃者が +300×本場、ツモは各家が +100×本場 を追加負担。供託は和了者が 1000×本数 を得る(`total` に含める。`payments` には含めない=卓上の供託であり誰かの支払いではないため)。

## 5. `payments` の形式

- **判断**: 以下の形式とする。
  - ロン: `[{ "from": "discarder", "amount": <ロン点数 + 300×本場> }]`
  - 子ツモ: `[{ "from": "dealer", "amount": <親の支払い> }, { "from": "non_dealer", "amount": <子の支払い> }, { "from": "non_dealer", "amount": <子の支払い> }]`
  - 親ツモ: `[{ "from": "non_dealer", "amount": <各子の支払い> }] × 3`
  - 各 `amount` は本場加算(+100×本場/人)を含む。`total = paymentsの合計 + 1000×供託`。
- **根拠**: 仕様は `payments` の要素形式を定義していない(「ツモ時は支払い内訳」のみ)。支払元の役割と金額が分かる最小の形式とした。ロン時も対称性のため1要素の内訳を返す。

## 6.1 `yaku` 配列の順序

- **判断**: 出力の `yaku` 配列の並び順は契約として規定しない(判定順による決定的な順序で出力するが、消費側は順序に依存しないこと)。ドラ(`dora` / `ura_dora` / `aka_dora`)は通常役の後に置く。
- **根拠**: 仕様に順序の規定がないため。

## 7. ダブル役満・役満の複合

- **判断**:
  - 単独でダブル役満となる特殊形(国士無双十三面待ち・四暗刻単騎待ち・大四喜・純正九蓮宝燈)を採用する。
  - 複数役満の複合を認め、**合計はダブル役満(基本点8000×2)を上限**とする(例: 大四喜+字一色+四暗刻 = 3倍相当だが2倍に丸める)。
  - 役満成立時、通常役およびドラは加算しない(出力の `yaku` にも含めない)。
  - 出力の `han` は `13 × 役満倍率(上限2)`。個々の役満エントリの `han` は単役満13・ダブル役満26。上限で丸めた場合、`yaku` 配列の翻合計が `han` を上回ることがある。
- **根拠**: 仕様は「役満の複合: あり(ダブル役満まで)」とのみ規定。一般的なルールに従い特殊形ダブルを採用しつつ、上限規定を合計倍率のキャップと解釈した。

## 8. 数え役満

- **判断**: 通常役+ドラの合計が13翻以上なら基本点8000(シングル役満相当)として計算する。出力の `han` は実翻数をそのまま返す(13に丸めない)。
- **根拠**: 仕様「数え役満: 13翻以上」。翻数自体は事実として保持し、点数のみ役満相当とするのが自然。

## 9. 符計算の詳細(仕様の付録C が TODO のため自定義)

- **判断**: 一般的な符計算を採用する。
  - 副底20符。門前ロン+10符。ツモ+2符(平和ツモを除く)。
  - 待ち: 嵌張・辺張・単騎+2符。両面・双碰は0符。
  - 雀頭: 三元牌+2符、自風+2符、場風+2符。**連風牌(自風=場風)の雀頭は加算で4符**(天鳳系ルールを採用)。
  - 面子: 明刻2/明槓8/暗刻4/暗槓16符、么九牌はそれぞれ2倍。ロン和了で完成した刻子は明刻扱い。
  - 総計を10符単位に切り上げ。
  - 七対子は25符固定。平和ツモは20符。門前平和ロンは30符。
  - **副露手で加符が0符(いわゆる喰い平和形)のロンは30符に引き上げる**(20符のままにしない)。
  - 嶺上開花のツモ符(+2)はあり。
- **根拠**: 付録C(符計算の詳細手順)が未作成のため、広く使われる標準(天鳳等)に従った。連風雀頭は2符とする流派もあるが、加算式(4符)の方が採用例が多いオンライン標準に合わせた。

## 10. 役満時の符の出力

- **判断**: 役満でも `fu` フィールドは数値で出力する。通常形の役満は通常の符計算値、七対子形(字一色)は25、国士無双は一律30とする。
- **根拠**: 出力契約が `fu` を必須としているが、役満の点数は符に依存しない。国士無双には標準的な符の定義がないため固定値30とした(点数へは影響しない)。

## 11. ドラの扱い

- **判断**:
  - `dora` / `uraDora` は「ドラそのもの」の牌種の配列。配列に同種が複数あれば(槓ドラ等)その回数分カウントする。
  - 赤5(`0m/0p/0s`)は照合上は5として扱う(ドラ指定 `5m` は手中の `0m` にもマッチする。`dora` に `0m` と書かれた場合も `5m` と同義に正規化)。
  - `uraDora` はリーチ(`riichi` / `double`)時のみ加算。リーチなしで `uraDora` が指定された場合は**エラーとせず無視**する(仕様の「リーチ時のみ有効」に従う)。
  - 赤ドラは手牌+副露中の `0m/0p/0s` の枚数分を `aka_dora` として加算する。
  - ドラは役ではない。ドラ以外に1翻もない手は `no_yaku` とする。
  - ドラがある場合、`yaku` 配列に `{"name":"dora","han":n}` / `{"name":"ura_dora","han":n}` / `{"name":"aka_dora","han":n}` の形で(枚数を han に集約して)含める。
- **根拠**: 仕様はドラの記法のみ定義し、出力への載せ方・赤との照合を未定義のため、一般的な計算機の慣行に合わせた。

## 12. 入力バリデーション(`invalid_input` の条件)

- **判断**: 以下を `invalid_input` とする。
  - JSON として不正、トップレベルがオブジェクトでない、必須フィールド(`hand` / `winTile` / `winType` / `seatWind` / `roundWind`)の欠落・型不一致
  - 不正な牌表記、不正な列挙値(`winType` / `seatWind` / `roundWind` / `riichi` / `situational`)
  - `hand` の枚数が §1 のどちらの形式にも合致しない
  - 完成形(手牌+副露+和了牌)で同種牌が5枚以上、または同色の赤5が2枚以上
  - 副露の構成が不正(チーが順子でない、ポンが3枚同種でない、カンが4枚同種でない、チーに字牌を含む等)
  - `winTile` が完成形の手牌に含まれない
  - 論理矛盾する状況: 一発なのにリーチなし / リーチなのに副露あり(暗槓は可) / 海底なのにロン / 河底なのにツモ / 嶺上なのにロン・槓なし・一発 / 槍槓なのにツモ / 海底+嶺上の同時指定 / 河底+槍槓の同時指定 / 天和・地和の条件不成立(天和=親・ツモ・副露なし・リーチなし、地和=子・ツモ・副露なし・リーチなし、両者の同時指定や他の状況役との併用は不可)
- **判断(寛容側)**: `melds` / `dora` / `uraDora` / `riichi` / `ippatsu` / `situational` / `honba` / `kyotaku` の省略はデフォルト値(空配列 / `"none"` / `false` / 0)で補完する。未知の追加フィールドは無視する。`situational` の重複はエラーとせず重複除去する。
- **根拠**: 仕様は `invalid_input` の条件を列挙していない。構造的不正と物理的・論理的に不可能な状態を不正とし、省略・冗長は寛容に扱うのが CLI として頑健と判断した。

## 13. `winTile` と赤5の照合

- **判断**: `winTile` と手牌の照合は牌種ベース(赤5と通常5は同種)で行う。赤ドラの枚数は手牌・副露に含まれる `0m/0p/0s` の枚数から数えるため、どの1枚を和了牌とみなすかは点数に影響しない。
- **根拠**: 和了牌が赤かどうかは役・符・点数のいずれにも影響しないため、厳密な区別は不要。

## 14. 高点法の同点判定

- **判断**: 複数解釈の比較は (1) 点数合計 → (2) 翻数 → (3) 符 の優先順で高い方を採用する。
- **根拠**: 仕様は「最も点数が高くなる解釈」のみ規定。同点時の決定性のため翻→符の順のタイブレークを設けた(一般的な高点法の慣行)。

## 15. 七対子の制約

- **判断**: 七対子は「7種の異なる対子」とし、同一牌4枚を2対子と数えることは認めない。副露(暗槓含む)がある場合は成立しない。二盃口形は七対子・通常形の両方で解釈し高点法で選択する。
- **根拠**: 標準ルールに従った。

## 16. 混全帯么九・純全帯么九と混老頭・清老頭の関係

- **判断**: チャンタ・純チャンは順子を1つ以上含む場合のみ成立とする(順子がない場合は混老頭・清老頭の領域であり複合しない)。
- **根拠**: 標準的な複合規則(混老頭とチャンタ系は複合しない)に従った。

## 17. 状況役と門前清自摸和の複合

- **判断**: 海底・嶺上のツモ和了でも門前なら `menzen_tsumo` を併記する。天和・地和は役満のため通常役は付かない(§7)。
- **根拠**: 標準ルールに従った。
